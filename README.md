### Intended Production Integration

In a production deployment:

1. Endee would be started using its provided `docker-compose.yml`.
2. Embeddings generated by this application would be sent to the Endee service.
3. Queries would be executed against Endeeâ€™s indexed vector store.
4. Top-k similar results would be returned to the backend.

### Current Implementation

For demonstration purposes, this project implements an abstraction layer called:

```
EndeeClient
```

This simulates interaction with the Endee service while preserving architectural alignment.

The abstraction layer ensures:

- Clean separation of concerns
- Replaceable storage backend
- Production-ready design
- Clear vector database interaction boundary

This design allows seamless transition to real Endee service integration.

### Deployment Note

Endee is designed to run as a standalone C++ vector database service using Docker.

For portability and ease of evaluation, this project demonstrates Endee integration through a clearly defined abstraction layer (`EndeeClient`) rather than requiring a full Docker-based deployment.

This ensures:

- The project runs immediately without infrastructure setup
- The vector database interaction boundary is clearly defined
- The system remains fully aligned with Endeeâ€™s architecture

In a production scenario, the `EndeeClient` would directly communicate with a running Endee service via its API.

## ğŸ§® How Vector Search Works

1. PDF text is chunked into fixed-size segments.
2. Each chunk is converted into a numerical embedding vector.
3. Vectors are stored in the EndeeClient abstraction.
4. User query is converted into an embedding.
5. Cosine similarity is computed between query and stored vectors.
6. Top-k most similar chunks are returned.

Vector similarity is the core retrieval mechanism.
